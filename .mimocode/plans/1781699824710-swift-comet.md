# Fix: Gallery Modal Stale Images in Production

## Root Cause

The issue is **server-side data caching** in production.

### How the data flows:

1. **Upload page** (`GalleryWrapper.jsx`): Fetches images **client-side** via `useEffect` → `getImages()` → always fresh ✅
2. **Add Service page** (`add-service/page.js`): Fetches images **server-side** at render time → `getImages()` → passed as `safeImages` prop to `ServiceFormWrapper` → `FileUploaderButton` → `AllFileGellary`

### Why it breaks in production:

- `add-service/page.js` is a **Server Component** that calls `getImages()` at page render time
- In production, Next.js **caches server-rendered pages** (static generation / ISR)
- When you upload an image and navigate to add-service, the page returns **cached HTML with stale `safeImages`**
- `revalidatePath("/dashboard")` in `galleryAction.js:40` only revalidates paths starting with `/dashboard`, but the add-service page may already be cached at build time or by Next.js data cache
- Locally it works because Next.js dev mode doesn't aggressively cache server components

### Key files:

| File | Role |
|------|------|
| `app/(dashboard)/dashboard/services/add-service/page.js:12-17` | Server component that fetches images once at render |
| `actions/galleryAction.js:49-63` | `getImages()` - no cache control |
| `app/(dashboard)/_components/gallery/AllFileGellary.jsx` | Modal that receives static `images` prop |
| `app/(dashboard)/_components/services/FileUploaderButton.jsx:28-33` | Passes `safeImages` directly to modal |

---

## Solution Options

### Option A: Client-side fetch in AllFileGellary (Recommended)

Make `AllFileGellary` fetch its own images from the server when opened, instead of relying on parent props.

**Changes:**
- `AllFileGellary.jsx`: Add client-side `getImages()` call when modal opens
- Keep `images` prop as initial/fallback data
- Add `no-store` or cache-busting to the fetch

**Pros:** Minimal changes, fixes the root cause for ALL places that use AllFileGellary
**Cons:** Adds an extra server call each time modal opens

### Option B: Add `fetch` cache: 'no-store' to getImages

In `galleryAction.js`, force `getImages()` to never cache by using `cache: 'no-store'` in the MongoDB query or adding `unstable_noStore()`.

**Changes:**
- `actions/galleryAction.js`: Add `import { unstable_noStore } from 'next/cache'` and call it in `getImages()`

**Pros:** Simple one-line fix
**Cons:** May affect performance if getImages is called frequently

### Option C: Force dynamic rendering on add-service page

Add `export const dynamic = 'force-dynamic'` to `add-service/page.js`.

**Changes:**
- `app/(dashboard)/dashboard/services/add-service/page.js`: Add dynamic export

**Pros:** Very simple
**Cons:** Only fixes this specific page; same issue exists in other pages (settings, projects, blogs)

---

## Recommended Approach: Option B (simplest, broadest)

Add `unstable_noStore()` to `getImages()` in `galleryAction.js`. This prevents Next.js from caching the result, ensuring fresh data on every call — both server-side and client-side.

### Files to modify:

1. **`actions/galleryAction.js`** - Add `import { unstable_noStore } from 'next/cache'` and call `unstable_noStore()` at the start of `getImages()`

### Verification:

1. Upload a new image on `/dashboard/upload-files`
2. Navigate to `/dashboard/services/add-service`
3. Open the gallery modal → new image should appear
4. Repeat for other pages using `AllFileGellary` (settings, projects, blogs)
